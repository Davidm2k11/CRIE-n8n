## Certification (§428 / §512 / §527)

Two levels:

1. **Per-Knowledge-Unit (§512):** atomic statement · valid category (R-05) ·
   authority assigned · evidence exists · citation exists (R-06) · duplicate
   check passed · quality score ≥ threshold. A unit that fails any check is left
   `Draft` and flagged `reviewRequired` (§513) — it never enters retrieval.
2. **Per-Document (§428 / §527):** repository structure · knowledge quality ·
   evidence quality · citation completeness · embedding availability · version
   integrity · transaction success. Emits the §527 certification object
   (`repositoryVersion`, `certified`, `certificationDate`, `qualityScore`).

## Quality Score (§511)

Deterministic 0.00–1.00 from OCR confidence, citation completeness, structural
quality, statement clarity, category confidence, and duplicate probability.
Weights are configurable (Principle 7); defaults live in the config registry.

## Versioning + Lineage (§519 / §522)

Each certified unit records `version`/`previous_version`, `created_by`,
`change_reason`, and a `lifecycle_state` (§518: Draft → Validated → Certified →
Deprecated → Archived). Superseded units are **Deprecated, never deleted**
(§522, §530). Every write allocates a repository version (§423).

## Integrity + Governance Gates (§524 / §525 / §427)

Before commit the writer rejects: knowledge without evidence, evidence without
citations, chunks without knowledge units, orphan embeddings, knowledge without
authority, invalid categories, and citations not keyed on evidence_id +
document_id (R-06). Any violation blocks the transaction and thus certification.

## Ownership (R-13 / §425)

`assertRepositoryOwnership` permits only Repository-owned objects (document,
metadata, knowledge unit, evidence, citation, chunk, embedding) and **refuses**
Compliance Result, Context Package, prompts, and AI conversations (§427, §54).
Chunks and embeddings are written here (generated by Ingestion, consumed — never
owned — by Retrieval).

## Transaction (§148 / §234)

All canonical objects are written in one transaction; any failure rolls back.
Partial writes are prohibited.

## Repository API (§365 / §56)

`createDocument`, `updateDocument`, `archiveDocument`, `getKnowledgeUnit`
(certified-only, §518), `searchMetadata`, `searchKnowledge` (certified-only),
`rebuildEmbeddings` (reserved triggers only, §523; enforces vector(1536), R-09),
`repositoryStatistics` (§529), `repositoryHealth` (§528). Responses use canonical
`{status, data | error}` contracts. The Repository is never bypassed (§608).

## Health Score (§528) + Statistics (§529)

Health is a weighted 0–100 score (Healthy / Degraded / Critical bands) over
certified-knowledge %, duplicate rate, missing citations, broken relationships,
embedding coverage, and extraction success. Statistics expose documents,
knowledge units, evidence, citations, chunks, embeddings, average quality,
repository size, average version, and health. Presentation as Supabase views in a
BI tool is **Sprint 8 (R-17)** and is not built here.
